Day 18 — Promises & async flow

Lesson

Open DevTools → Console, or run node day-18/js/lesson.js (takes ~20s — timers). Source: ../js/lesson.js.

1 Sync vs async

Synchronous code runs top-to-bottom. setTimeout(fn, 0) still runs later on the event loop — so you often see 1, 3, 2 instead of 1, 2, 3.

../js/lesson.js — Topic 1

2 Callbacks

Pass a function to be called “when done”. Node-style: (err, data) as the first arguments to the callback.

../js/lesson.js — Topic 2 (fetchUserCb)

3 Callback hell

Nested callbacks (“pyramid of doom”): hard to read, repeated error handling, hard to extend — this motivated Promises.

../js/lesson.js — Topic 3

4 Promise states

pending → work in progress; fulfilled / rejected → settled (terminal). delay(ms) wraps setTimeout in a Promise.

Follow-along: logging p right after new Promise shows a pending promise; .then runs when the executor calls resolve(42).

../js/lesson.js — Topic 4

5 .then / .catch / .finally

Chains stay flat. Returning a value passes it to the next .then; returning a Promise is awaited by the chain. .finally runs on both success and failure.

../js/lesson.js — Topic 5 (fetchUserPromise)

6 Promise combinators

Promise.all (fail-fast), allSettled (never rejects), race (first settled), any (first fulfilled). The runnable lesson uses mock timers instead of real fetch so it works offline in Node.

../js/lesson.js — Topic 6

7 Promise.resolve / Promise.reject

Shortcuts for already-settled values. Pattern: getUser(id, cache) returns Promise.resolve(cache[id]) or a real async fetch — callers treat both the same.

../js/lesson.js — Topic 7
